【四、算法主线】
请严格围绕以下 8 个模块组织代码。

========================
模块1：问题场景与系统假设
========================
- 多车辆、多任务、在线动态任务分配
- 任务是纯访问型，无时间窗
- 每个任务只能由一个车辆执行
- 所有任务最终必须完成，除非被取消
- 车辆异构性体现在速度与容量
- 车辆采用 Dubins 风格约束，但第一版不要求严格轨迹优化，仅在代价中体现转向修正
- 环境变化不直接触发任务层重拍卖，只影响代价估计精度

========================
模块2：目标与约束
========================
- 总目标：最小化所有车辆任务执行时间总和
- 单个任务加入某车辆序列后的代价 = 增量执行时间
- 增量时间 = 避障路径长度 / 车辆速度
- 任务默认追加到当前任务序列末端
- 容量不足的车辆不能竞拍该任务

========================
模块3：快速代价预估器
========================
对车辆 i 和任务 j，快速预估长度定义为：

L_hat_ij = d_ij + lambda_psi * R_i * abs(delta_psi_ij) + lambda_rho * d_ij * rho_ij

对应预估时间代价：
c_hat_ij = L_hat_ij / v_i

其中：
- d_ij：当前序列末端到任务点的欧氏距离
- delta_psi_ij：当前末端航向与目标方向偏差
- R_i = v_i / omega_i_max
- rho_ij：末端到任务点的走廊区域内障碍密度
- 默认参数：
  - lambda_psi = 0.15
  - lambda_rho = 1.20

请实现：
1. 基于当前车辆任务序列末端计算候选任务代价
2. 障碍密度 rho_ij 通过“连接末端与任务点的矩形走廊区域”内障碍占比估计
3. 速度差异进入时间代价

========================
模块4：竞拍机制
========================
- 每辆车在当前可行任务集中，选择 c_hat 最小的那个任务进行竞拍
- 同一任务由 c_hat 更小的车辆赢得
- 获胜任务先进入 tentative 状态
- 任务拍得后默认追加到该车辆任务序列末端
- 新任务也按当前序列末端进行竞拍

========================
模块5：邻域协商与任务状态同步
========================
- 不使用经典拍卖价格
- 每个任务维护本地记录：
  (winner, bid, status, version)
- 状态包括：
  unassigned, tentative, verifying, locked, withdrawn, canceled
- 车辆只与邻居进行信息交换
- 邻居定义建议使用通信半径
- 同步采用离散时步 + 同步轮次方式
- 冲突消解顺序：
  1. 状态优先级
  2. version 更大
  3. bid 更小
  4. vehicle id 更小
- 建议默认参数：
  - H = 2 （连续稳定轮数）
  - R_max = 5 （单次协商最大轮数）

请实现一个轻量级邻域协商层，不要做复杂分布式一致性理论。

========================
模块6：高精度代价校对
========================
对于 tentative 中标任务，执行高精度校对：
1. 用高分辨率 A* 计算实际避障路径长度 L_astar_ij
2. 计算高精度代价：

c_tilde_ij = (L_astar_ij + lambda_psi * R_i * abs(delta_psi_ij)) / v_i

3. 定义低估率：
e_under = (c_tilde_ij - c_hat_ij) / c_tilde_ij

4. 若 e_under <= epsilon，则通过校对并锁定任务
5. 否则任务撤回并进入模块7

默认参数：
- epsilon = 0.25

A* 要基于高分辨率栅格地图，并考虑障碍膨胀（车辆半径 + 安全距离）。

========================
模块7：撤回与局部再拍卖
========================
- 校对失败时，仅更新失败车辆对该任务的本地拍卖代价：
  c_hat_ij <- c_tilde_ij
- 不要加入额外惩罚项
- 任务状态更新为 withdrawn
- 任务重新加入局部待拍卖池
- 触发局部再拍卖
- 若更新真实代价后该车辆再次中标，则直接接受，不再因同一原因再次撤回

请实现这个逻辑闭环。

========================
模块8：在线动态事件处理
========================
只对任务侧事件做任务分配层更新：
1. 新任务到达
2. 任务取消

环境变化：
- 只更新地图或影响代价估计
- 不直接触发任务层重拍卖

任务取消处理规则：
1. 若任务尚未 locked：
   - 直接改为 canceled
   - 从待拍卖池中移除
2. 若任务已 locked 但尚未执行：
   - 从所属车辆的后续任务序列中删除
   - 恢复车辆剩余容量
   - 做本地序列拼接更新
   - 不做全局重算
3. 已完成任务不取消
